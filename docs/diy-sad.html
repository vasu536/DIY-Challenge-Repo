System Overview & Goals

1.1 Competition Context

The CAT Robotics Culture Club DIY Robot Challenge 2026 is a two-day autonomous vehicle race held outdoors at the Tech Center Bldg. A in October 2026. Teams build custom robots and compete on two courses: a speed course (3 laps) and an obstacle course (2 laps). Scoring is time-based with cumulative time penalties for skipped obstacles.

1.2 Primary Goals

MVP Goal — Start to Finish

Complete both laps of the obstacle course from start to finish, crossing all or most obstacles correctly, within the 10-minute heat window. Minimum viable: crossing the majority of obstacles using the correct behavior (push-through vs navigate-around) without corrupting the map for lap 2.

Stretch Goal — Competitive Score

Zero obstacle skips, consistent 2-lap completion under ~4 minutes per run, with sub-3cm localization accuracy at all critical waypoints (tunnel entry, narrow passages). Reliable lap-2 consistency enabled by RTK drift correction.

Critical — Wheel Encoder Decision
Wheel encoder data from CAN is intentionally excluded from the EKF. FWD under load on gravel causes front-wheel slip, and differential slip on turns corrupts heading. FAST-LIO2 LiDAR-inertial odometry (0.1–0.3% drift) outperforms CAN wheel odometry even on ideal surfaces. Adding bad data to an EKF makes it worse, not better.

Compute Architecture — STM32 & Jetson Nano

3.1 Responsibilities Split

STM32 — Real-Time Layer

Responsibilities:

  • CAN bus driver — translates /cmd_vel ROS2 topic → P-CAN motor commands
  • E-Stop handler — wired (course) and wireless (team) inputs, forces /cmd_vel = 0 within 1 second
  • Fail-safe: if wireless signal lost >1s, autonomously stop
  • micro-ROS bridge over UART — exposes STM32 to ROS2 graph on Jetson

Why STM32 and not Jetson directly? Hard real-time guarantees. The E-stop response requirement (<1s, Rule 1.2.7) and motor control loop require deterministic timing that Linux/ROS2 on Jetson cannot guarantee under load.

Jetson Nano — ROS2 Humble Stack

All software layers:

  • All sensor drivers (LiDAR, camera, IMU, GPS)
  • FAST-LIO2 LiDAR-inertial odometry
  • robot_localization EKF (local + global)
  • PCL obstacle classification pipeline
  • Nav2 (planner + controller + costmap)
  • Mission state machine + behavior switcher
  • Visual start detector
  • AprilTag detection (calibration runs)

Performance Watch
Jetson Nano has 4GB RAM and a 128-core Maxwell GPU. At 75–85% CPU, there is limited headroom. Profile early — if FAST-LIO2 + Nav2 together exceed 90%, consider: (1) reducing Nav2 costmap resolution, (2) dropping FAST-LIO2 to 10Hz output, or (3) upgrading to Jetson Orin Nano which is pin-compatible.

Mode Switch
A mode_switch parameter controls whether /cmd_vel originates from joy_node (JOYSTICK mode) or Nav2 (AUTONOMOUS mode). This is enforced via a topic multiplexer node. During competition heats, the joystick driver is disabled by parameter — not just disconnected — to prevent accidental override.

4.2 Localization Layer

FAST-LIO2 — LiDAR-Inertial Odometry (Front-End)

FAST-LIO2 is the primary odometry source. It tightly couples the Hesai QT64 pointcloud with the dedicated IMU to produce high-rate (50Hz), low-latency ego-motion estimates without relying on wheel odometry.

Purpose:  Feeds Nav2 local controller — needs low latency
Frame:    odom → base_link

navsat_transform + EKF2 (Global Frame)

Inputs:   /odometry/filtered    (from EKF1)
          /gps/fix              (RTK GNSS, 5–20Hz, ~2cm accuracy)

Output:   /odometry/global      (globally referenced, drift-corrected)
Purpose:  Provides drift-free global pose for Nav2 global planner
          Corrects accumulated EKF1 drift at every GPS update
Frame:    map → odom (corrects the transform continuously)
Why Two EKF Instances?
EKF1 runs at high rate (50Hz+) in the local odom frame — this is what the Nav2 MPPI controller uses for tight trajectory tracking. EKF2 fuses GPS corrections at lower rate but maintains global consistency. Separating them prevents GPS latency (~100ms) from destabilizing the local controller.

AprilTag Pose Corrections (Calibration Sessions Only)

During pre-competition test runs in your own track, AprilTags at known surveyed positions provide full 6DOF ground truth (position + orientation). This is used to:

  • Profile FAST-LIO2 drift per track section
  • Auto-tune EKF1 process noise Q and measurement noise R matrices (Innovation-based Adaptive EKF)
  • Provide heading ground truth that single-antenna RTK cannot supply at low speed

AprilTags are NOT placed on the competition course — this is purely a calibration and parameter tuning tool used in your own test environment before competition day.

4.3 Perception Layer

PCL Obstacle Classifier Node

Custom ROS2 node consuming /hesai/points. Runs after each LiDAR scan, publishes classification results to the mission layer and updates the costmap accordingly.

Pipeline:
  Input: /hesai/points (PointCloud2, ~20Hz)

  Step 1: Ground plane removal
    → RANSAC plane fitting (max 3 iterations, threshold 0.05m)
    → Remove all points within 5cm of ground plane
    → Output: above-ground pointcloud

  Step 2: Euclidean clustering
    → pcl::EuclideanClusterExtraction
    → cluster_tolerance: 0.15m
    → min_cluster_size: 20 points
    → max_cluster_size: 5000 points

  Step 3: Map differencing (day-of scan only)
    → Compare cluster centroids vs prior_map.pcd
    → Points NOT in prior map = new obstacles
    → Filters out track walls, curbs (already in prior)

  Step 4: Per-cluster classification
    → Compute: bounding box, point density, vertical extent,
               horizontal spread, depth-image consistency (D435i)

    CHIMES:    thin vertical clusters, sparse density,
               height > 0.3m, width < 0.05m,
               OR D435i shows noisy/sparse depth in same FOV
               → Behavior: PUSH_THROUGH or BLIND_DRIVE_MODE

    MOVABLE:   isolated cluster, dense, horizontal cross-section
               similar in all axes (box-like), free space around it
               → Behavior: NAVIGATE_AROUND

    PASSAGE/   no obstacle cluster in forward frustum,
    TUNNEL:    bounded walls detected on both sides
               → Behavior: PUSH_THROUGH (enter, use dead-reckoning)

    WALL:      dense, large cluster spanning full height,
               no navigable gap detected
               → Behavior: stop, re-plan global path

  Output: /obstacle_class (custom msg: zone_id, classification, confidence)
          → consumed by Mission State Machine and Costmap Manager

Costmap Manager

LayerSourcePurpose
Static mapmap_server (prior map)Track boundaries, known walls — always present
Voxel layerLive /hesai/pointsDynamic obstacle inflation
Temporal decayVoxel layer parameterChimes appear/disappear — decay prevents permanent inflation
Inflation layerAll occupied cellsRobot footprint expansion for collision avoidance

The Behavior Mode Switcher can dynamically disable the voxel layer during PUSH_THROUGH and BLIND_DRIVE_MODE, preventing chimes or tunnel features from being treated as walls. The layer is re-enabled after the zone is cleared by encoder distance or RTK position.

4.4 Navigation Layer

Global Planner — Smac Hybrid-A*

Chosen over NavFn/Dijkstra because it natively understands non-holonomic constraints — the FWD robot cannot drive sideways, and has a minimum turning radius. Smac Hybrid-A* plans kinematically feasible paths that respect these constraints.

The global path is pre-computed from the centerline waypoints extracted from the prior map. During a run, the global planner is only re-invoked if an obstacle forces a significant deviation (e.g., a WALL classification).

Local Controller — MPPI

Model Predictive Path Integral controller. Samples thousands of trajectory rollouts, weights by cost, produces smooth commands. Better than DWB for:

  • Dynamic obstacle response (movable objects)
  • Smooth velocity profiles on ramp + gravel
  • Maintaining heading through narrow passages

map_server — Serving the Prior Map

Serves the obstacle-free prior map as the Nav2 static layer. Also loads waypoints.yaml (centerline waypoints) and zone_config.yaml (obstacle zone boundaries and behaviors). These three files are generated offline and uploaded to the robot before each competition heat.

Files loaded at startup:
  /maps/static_map.yaml        ← 2D occupancy grid from prior map
  /maps/prior_map.pcd          ← 3D pointcloud for FAST-LIO2 init
  /config/waypoints.yaml       ← pre-computed centerline waypoints
  /config/zone_config.yaml     ← obstacle zone entry/exit + behavior
  
  Variants: lab_test.yaml  |  competition_day.yaml

4.5 Mission Layer

Visual Start Detector

Monitors /camera/color/image_raw from D435i. Detects the course-provided visual start signal (Rule 2.5.1). When detected, publishes a /start_signal topic to transition the state machine from WAIT to NAVIGATE. A 5-second manual override is available per Rule 3.4.1 but incurs a time penalty.

Mission State Machine

States:
  WAIT_FOR_START_SIGNAL
    → monitors /start_signal
    → on detection: transition to NAVIGATE_TO_WAYPOINT

  NAVIGATE_TO_WAYPOINT
    → sends next waypoint goal to Nav2 action server
    → monitors /odometry/global for zone entry proximity
    → on zone_entry_threshold reached: transition to OBSTACLE_ZONE_ENTRY

  OBSTACLE_ZONE_ENTRY
    → looks up zone_config.yaml for current zone_id
    → reads PCL classification from /obstacle_class
    → resolves final behavior (config default OR live classification)
    → transition to BEHAVIOR_SWITCH

  BEHAVIOR_SWITCH
    → activates selected BehaviorMode (see below)
    → on zone_exit (RTK position OR encoder distance): restore NORMAL_NAV
    → transition back to NAVIGATE_TO_WAYPOINT

  LAP_COMPLETE
    → increments lap counter
    → if laps == 2: transition to STOP
    → else: restart from first waypoint

  STOP
    → publish /cmd_vel = 0
    → signal STM32 to hold brakes

Behavior Mode Switcher

Narrow Passage Critical Requirement
Track min width = 20". Robot width = 16". Only 2" clearance each side. At 2" = 5cm, you need lateral accuracy better than 4cm at passage entry. RTK + pre-approach RTK correction achieves this. FAST-LIO2 alone may not in worst case. Prioritize RTK fix before every narrow passage.

Get this right in week 1 of development — it is the hardest bug to fix last-minute.

5.3 Loop Closure Strategy

FAST-LIO2 has no loop closure. For this course, the strategy is:

  1. Primary: RTK GPS provides continuous global corrections via EKF2. On open-sky portions of the course, drift is continuously reset to ~2cm. This handles the lap-2 drift problem automatically for most of the course.
  2. Secondary (if needed): Add Scan Context++ to FAST-LIO2 (FAST_LIO_SLAM repo). This detects that the robot has returned to a previously visited location (lap 2 start ≈ lap 1 start) and fires a loop closure via GTSAM pose graph. Evaluate necessity after the August path-following demo — if RTK drift between laps is <5cm, this is not needed.

Offline Pre-Competition Pipeline

All offline processing happens before competition heats. The robot enters each heat already knowing the map, the waypoints, and the obstacle behaviors — it is not discovering anything from scratch during the run.

6.1 Phase 1 — Prior Map Recording

When: Weeks before competition, clean track (no obstacles placed yet)

  • 1
    Joystick drive — 2 full laps at 0.3 m/s Activate joy_node + teleop_twist_joy in JOYSTICK mode. Drive slowly and steadily — consistent speed reduces IMU disturbance and improves scan matching quality. Two laps are required so LIO-SAM's loop closure fires on the second lap, producing a globally consistent map.
  • 2
    Record full rosbag ros2 bag record /hesai/points /imu/data /gps/fix /camera/color/image_raw /tf /tf_static
    Store safely — this bag is irreplaceable until you can re-drive the clean course.
  • 3
    Offline LIO-SAM processing Play the rosbag through LIO-SAM in offline mode. LIO-SAM uses the GPS factor graph to maintain global consistency. The second-lap loop closure corrects any first-lap drift. Output: prior_map.pcd (globally consistent 3D pointcloud) and trajectory.pcd.
  • 4
    2D map extraction Run octomap_server on prior_map.pcd to generate a 2D occupancy grid (static_map.pgm + static_map.yaml). This becomes Nav2's static costmap layer.
  • 5
    Centerline extraction Apply Voronoi skeletonization to the 2D occupancy grid to extract the track centerline. Convert to Nav2 waypoint format → waypoints.yaml. Manually verify in RViz that waypoints stay within track boundaries.
  • 6
    Zone boundary annotation Using the course layout (provided per Rule 3.2 before racing), manually annotate obstacle zone entry/exit coordinates in the map frame → zone_template.yaml. One zone per obstacle. This file has no behavior values yet — only boundaries.

6.2 Phase 2 — Day-Before Obstacle Classification

When: Day before competition, after obstacles are placed on course

  • 1
    Slow scan lap with obstacles present Drive the course (joystick or slow autonomous) at 0.2 m/s. Record rosbag. RTK and QT64 must be active. Purpose: capture the full course with obstacles in place.
  • 2
    Map differencing Subtract prior_map.pcd from the day-of scan pointcloud using a nearest-neighbor threshold (0.1m). Points remaining after subtraction are new obstacles. This is far more reliable than trying to detect obstacles from scratch — the prior map filters out all permanent track features automatically.
  • 3
    Zone classifier For each annotated zone from zone_template.yaml, extract the obstacle clusters within that zone's bounding box and run the classification pipeline. Output: behavior assignment per zone.
  • 4
    Generate and review zone_config.yaml Merge zone boundaries (from template) + behaviors (from classifier) → zone_config.yaml. Open in RViz, visually verify each zone's classification makes sense against the pointcloud. Override any wrong classifications manually.
  • 5
    Upload and verify Deploy zone_config.yaml, static_map.yaml, waypoints.yaml to robot. Start the full stack. Verify in RViz that the static map loads correctly, waypoints display on the centerline, and zone boundaries highlight correctly. Fix any TF or file path issues now, not race morning.

6.3 AprilTag Calibration & EKF Auto-Tuning

When: Pre-competition test sessions in your own test track (weeks before competition)

AprilTags at known surveyed positions in your test track provide full 6DOF ground truth for system characterization. They are NOT placed on the competition course.

What You Get from AprilTag Ground Truth
  • Per-section drift profile of FAST-LIO2
  • Heading accuracy (RTK single-antenna poor at low speed)
  • IMU contribution to drift isolated by speed comparison
  • Systematic error patterns (e.g., always drifts left after right turn)
Innovation-Based Adaptive EKF Tuning

Use tag observations to back-calculate optimal Q (process noise) and R (measurement noise) matrices for robot_localization. The Innovation sequence (difference between predicted and observed tag pose) contains all information needed to tune these matrices automatically. Result: EKF uncertainty estimates match real-world uncertainty — tighter, more accurate fusion.

AprilTag calibration workflow:
  1. Place 8–10 tags at known surveyed positions (measure with RTK)
  2. Run 10+ laps at competition speed, record rosbag
  3. Post-process: compute FAST-LIO2 pose at each tag observation
  4. Compare to ground truth → drift vector per tag per run
  5. Fit EKF Q/R matrices using innovation covariance method:
       R_optimal = mean(innovation * innovation^T) over all tag observations
       Q_optimal = estimated from residual between filter and ground truth
  6. Save to competition.yaml
  7. Re-run 5 laps → verify improvement (drift should reduce 30–60%)

Lap 2 Consistency — Movable Objects
If the robot nudges a movable obstacle in lap 1, it changes position in the live costmap for lap 2. The prior map still shows the original position — the map diff for lap 2 will show the object has moved. The PCL classifier and mission layer must handle this gracefully: re-classify at zone entry for each lap, do not assume lap-2 obstacle positions are identical to lap-1.

Robot may use Ackermann steering until stop1.2.9STM32 zeroes drive velocity but allows steering to hold — prevents jackknifing
Critical Design Principle
The E-Stop path must NEVER pass through ROS2 or the Jetson. If the Jetson crashes, freezes, or is overloaded, the E-Stop must still work. The STM32 handles E-Stop independently. ROS2 on Jetson has no role in safety-critical stopping.

8.3 Recovery After E-Stop

When E-Stop is released and the robot resumes (Rule 3.7.2):

  • FAST-LIO2 re-localizes against the prior map using the current LiDAR scan — instant, no manual initialization needed
  • RTK GPS resumes providing global pose corrections within ~30 seconds of reacquiring fix
  • Mission state machine resumes from current state — no need to restart the full run
  • If the robot was repositioned by a team member (Rule 3.7.2), publish a manual pose estimate in RViz before resuming

Sensor Calibration Procedures

Priority Order
Calibration must be done in this order. Each step depends on the previous. Do NOT skip to software development until calibration is complete — all odometry, SLAM, and obstacle detection performance depends on accurate calibration.
  • 1
    IMU Intrinsic Calibration — imu_utils
    Place robot on stable surface. Record IMU at rest for 2–3 hours. Run imu_utils to compute Allan variance → outputs gyroscope and accelerometer noise density, bias instability, and random walk parameters. Enter these values into FAST-LIO2's config.yaml. This is the single most impactful calibration for LiDAR-IMU odometry quality. 
    Output parameters needed by FAST-LIO2:
  gyr_cov:   gyroscope noise density (rad/s/√Hz)
  acc_cov:   accelerometer noise density (m/s²/√Hz)
  b_gyr_cov: gyroscope bias instability
  b_acc_cov: accelerometer bias instability
  • 2
    IMU–LiDAR Extrinsic Calibration — lidar_imu_calib
    Drive the robot in figure-8 patterns (or random aggressive motion) while recording. The tool finds the 6DOF rigid-body transform between IMU frame and LiDAR frame. A 1° rotation error causes ~2cm/m positional drift in FAST-LIO2. Output: 4×4 transformation matrix → URDF and FAST-LIO2 config.
  • 3
    Camera–LiDAR Extrinsic Calibration — cam_lidar_calib
    Place a calibration target (checkerboard or AprilTag board) visible to both D435i and QT64. Collect observations from multiple angles. Output: transform between camera_link and lidar_link. Required for fusing D435i depth with QT64 pointcloud in the obstacle classifier.
  • 4
    GPS Antenna Lever Arm
    Physically measure the 3D offset from the GPS antenna phase center to the robot's base_link origin (typically robot geometric center at ground). Enter into LIO-SAM/EKF2 config as extrinsicTrans. An uncompensated 10cm lever arm causes systematic heading error.
  • 5
    IMU–LiDAR Time Synchronization Verification
    Record a rosbag while rotating the robot quickly. Visualize the pointcloud — if scans appear "smeared" or curved, there is a time offset between IMU and LiDAR timestamps. Fix via hardware trigger sync (if IMU supports it) or software timestamp correction in the driver. Target: <1ms offset.
  • 6
    EKF Parameter Tuning — AprilTag Calibration Sessions
    Set up AprilTag test track (8–10 tags at surveyed positions). Run 10+ laps. Use Innovation-based Adaptive EKF method to optimize Q and R matrices for robot_localization. Save to competition.yaml. Verify improvement with 5 more laps. Target: <5cm drift per lap in EKF1.
